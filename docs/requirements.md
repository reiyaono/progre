# 要件定義書

> OVLD（筋トレメモアプリ）を参考に、機能を自作するWebアプリ。
> ※ お手本アプリの画像・ロゴ・文言・配色・コードは流用せず、機能とUXのみ参考に独自実装する。

最終更新: 2026-06-09 / ステータス: **要件定義 完了（レビュー反映済み・設計フェーズへ）**

---

## 1. 背景・目的
- **何を作るか**: 筋トレの記録・履歴・進捗を管理するWebアプリ。
- **なぜ作るか**: 自分の筋トレ記録をメモし、漸進性過負荷（progressive overload）＝「先週の自分を超えたか」を可視化して継続を支援する。
- **学習目的**: Claude Code の skills / subagents(Task) / workflow / plan mode を実地で習得する。
- **プロダクトの核**: 記録の手軽さ × 「OVERLOADED（過負荷を達成したか）」の分析演出。前回比較・進捗管理が便利ポイント。

## 2. ターゲット・利用環境
- プラットフォーム: **Webアプリ**（スマホブラウザ前提、PWA化は後で検討）
- 利用シーン: ジム内でセット間に素早く入力。片手・数タップで完了したい。
- 想定ユーザー: まずは自分専用（ただし認証ありのマルチユーザー構造で作る）。

### 3.1 記録の階層（重要・確定）
```
1日（Workout: 日付ごと）
  └─ 種目エントリ（WorkoutExercise: その日やった種目, メモを持つ）
       └─ セット（WorkoutSet: 重量 × 回数, インターバル秒）
```
※ ドラフト初版は「1日1種目」になっていたが、実際は **1日 → 複数種目 → 複数セット** の3階層。

### 3.2 記録項目（確定）
- セット単位: **重量(kg)**, **回数(reps)**, **インターバル(秒)**
- 種目エントリ単位: **メモ**
- 単位: kg のみ（lb切替はLater）
- RPE: **Later**（MVP対象外）

### 3.3 「便利・特徴的」ポイント（再現したい価値）
- **前回比較**: 同じ種目の前回記録を即参照／自己ベスト超えが分かる。
- **進捗管理**: トップセット推移・部位別ボリューム推移・推定1RM/Max推移のグラフ。

## 5. 機能要件（MVP詳細）

| ID | 機能 | 概要 | 受け入れ条件（Done の定義） |
|----|------|------|------------------------------|
| F-01 | 認証 | ID/PASSで登録・ログイン。データはユーザーに紐づく | ・新規登録／ログイン／ログアウトができる<br>・未ログイン時は記録画面に入れずログインへ誘導<br>・他ユーザーのデータが見えない |
| F-02 | カレンダー | 月カレンダー。日付タップでその日のワークアウトへ。各日に実施部位の色ドット | ・前月/翌月/今日へ移動できる<br>・記録のある日に、実施した**部位の色ドット**が出る<br>・日付タップでその日のワークアウト画面へ遷移 |
| F-03 | 種目マスタ | 部位・トレーニング方法・色を持つ種目を管理。部位/方法もユーザー編集可 | ・種目のCRUDができる（名前/部位/方法を指定）<br>・部位カテゴリのCRUD＋**部位ごとの色設定**ができる<br>・トレーニング方法のCRUDができる<br>・部位タブで種目を絞り込める |
| F-04 | 記録 | カレンダーの日付に対し、複数種目・各複数セットを記録 | ・その日に種目を複数追加できる<br>・種目ごとにセット（重量×回数＋インターバル）を追加/削除/編集できる<br>・種目ごとにメモを残せる<br>・**前回同種目の記録**を参照できる<br>・保存後カレンダーに色ドットが反映される |
| F-05 | ダッシュボード | 下記4指標を表示 | ・**部位別ボリューム推移**（週次Σ重量×回数）<br>・**種目別最大重量推移**<br>・**週次OVERLOADEDレポート**（今週vs先週の部位別ボリューム前週比）<br>・**推定1RM / Max推移**（Epley式）<br>・期間/部位/種目で対象を切替できる |

### 5.1 計算ロジック（定義）
- **ボリューム** = Σ(weight × reps)（種目/部位/日/週で集計）
- **週の区切り** = **日曜始まり**（確定）
- **推定1RM** = Epley式 `1RM = weight × (1 + reps/30)`。種目の各セット最大値を採用
- **トップセット** = その種目エントリ内で最大重量のセット
- **OVERLOADED判定** = ある部位の「今週ボリューム > 先週ボリューム」で達成扱い

## 6. 非機能要件
- 性能: 記録入力は数タップ/数秒で完了。一覧/グラフ表示は体感1秒以内。
- オフライン: **対応しない**（Webオンライン前提）。
- データ保存: **クラウドDB**に保存。
- 認証: ID/PASS（メール＋パスワード想定）。
- セキュリティ: パスワードはハッシュ保存。ユーザー間のデータ分離（行レベル）。

## 7. データモデル（たたき台 v2）
```
User           (id, login_id/email, password_hash, created_at)

BodyPart       (id, user_id, name, color, sort_order)        -- 部位カテゴリ（色を持つ・編集可）
TrainingMethod (id, user_id, name, sort_order)               -- 自重/マシン/フリーウェイト等（編集可）
Exercise       (id, user_id, name, body_part_id[NOT NULL], training_method_id[NULL可], sort_order, is_archived)

Workout         (id, user_id, date)                          -- 1ユーザー×1日で1件
WorkoutExercise (id, workout_id, exercise_id, sort_order, memo)   -- その日やった種目（メモ保持）/ UNIQUE(workout_id, exercise_id)
WorkoutSet      (id, workout_exercise_id, set_no, weight, reps, interval_sec)
```
- 色は **BodyPart に保持**（カレンダーのドット＝その日の WorkoutExercise→Exercise→BodyPart.color の集合）。
- `body_part_id` は **NOT NULL**（カレンダーの色ドットが部位色に依存するため）。`training_method_id` は任意。
- **削除方針（確定）**: マスタ系（Exercise/BodyPart/TrainingMethod）は**論理削除**（`is_archived`／無効化）のみで物理削除しない＝過去記録・集計・ドットを保持。記録系（WorkoutExercise/WorkoutSet）は物理削除可。種目エントリを全削除した日の `Workout` は自動削除しドットも消す。
- 初期データ: 部位（胸/肩/背/脚/腹/有酸素＋色）＋方法（自重/マシン/フリーウェイト）＋**代表種目を数個**（例: ベンチプレス等）をシードし、以後ユーザー編集可。

## 8. 技術選定（候補・設計フェーズで確定）
> 設計フェーズで Plan agent に複数案を比較させる。
- フロント: **Nuxt** もしくは Vue.js（`.nuxtrc` あり ＝ Nuxt慣れと推定）
- DB/バックエンド: クラウドで無料枠 → **Supabase（Postgres＋Auth＋RLS）** が有力候補
- ホスティング: 無料枠 → Vercel / Cloudflare Pages 等
- グラフ: Chart.js / ECharts など

## 9. 詳細決定事項（要件レビュー反映）

### 9.0 削除・参照整合（高1）
- マスタ系は論理削除（`is_archived`）のみ。記録系は物理削除可。最後の種目を消した日の `Workout` は自動削除（§7 削除方針に詳細）。

### 9.1 入力バリデーション（高2）
- **weight**: 0以上、**0.25刻み**の小数（自重種目は0kg可）。上限ガード `weight ≤ 999`。
- **reps**: 1以上の整数。上限 `reps ≤ 99`。
- **interval_sec**: 任意（空＝未記録）。0以上の整数。
- 異常値・桁あふれは保存前にバリデーションで弾く。

### 9.2 ダッシュボード表示仕様（高3）
- **既定期間: 直近12週**（期間は切替可）。
- 推移系（部位別ボリューム／種目別最大重量／推定1RM）は**折れ線グラフ**。
- OVERLOADEDレポートは**部位カード**で前週比差分テキスト＋達成部位に**OVERLOADEDバッジ**（アニメ等の演出はLater）。
- 先週データが無い部位は **「New（比較対象なし）」** 表示。全体0件は**空状態メッセージ＋記録導線**。

### 9.3 認証・画面（中4）
- 画面に **ログイン / 新規登録** を追加。パスワードは **8文字以上**。
- メール確認は**オフ**（個人利用のため即ログイン可）。パスワードリセットは **Later**。
- 未ログイン時はルートミドルウェアでログインへ誘導（F-01）。

### 9.4 「前回記録」参照の定義（中5）
- 「前回」＝**編集対象日より過去で同種目を含む直近の Workout**。
- 表示は**前回トップセット（重量×回数）＋全期間の自己ベスト重量**。
- 自己ベスト超え判定は**最大重量**で行う。

### 9.5 並び順・初期表示（中6）
- 種目は `sort_order` 昇順（同値は作成順）。
- `set_no` は保存時に1から振り直し（**歯抜けなし**）。
- カレンダーは常に**今日を含む当月**を初期表示。
- 種目マスタは**先頭の部位タブ**を初期選択。

### 9.6 日付・タイムゾーン（中7）
- 日付・「今日」・週境界の判定は **JST（Asia/Tokyo）固定**。
- `Workout.date` は JST の暦日として扱う。

### 9.7 カレンダーのドット表示（中8）
- その日に実施した**部位のユニーク色ドットを最大4個**まで横並び。超過分は **「+n」** 表示。

### 9.8 種目マスタの一意性・空状態（低9・11）
- `body_part_id` は **NOT NULL 必須**（色ドットの前提）。`training_method_id` は任意。
- 同名種目はユーザー内で**警告のみで許容**（一意制約は課さない）。
- 新規登録時に代表種目を数個シードし、各画面の空状態に一言メッセージ＋次アクション導線を置く。

## 10. 決定事項（旧・未決事項）
- **週の開始曜日**: **日曜始まり**で確定。
- **同一部位内の種目色分け**: 不要。色は**部位単位で統一**（同日に胸2種目なら同色ドットでOK）。
- **1日に同一種目の複数回記録**: 許容しない。同一種目は1エントリにまとめ、回数違いは**セットで表現**する。
  - → データ制約: `WorkoutExercise` は `(workout_id, exercise_id)` で一意。
- **過去日付の遡り入力・編集**: **必要**（任意の日付の記録を後から追加・編集できる）。
